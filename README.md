#大一面向对象程序设计期末答辩——c++扫雷游戏

#答辩问题

1. 这个项目整体怎么设计的？

 用 WinAPI 做窗口和消息循环，核心逻辑放在 Cell 和棋盘数组里，界面、逻辑、图片加载分开处理。 
 
2. 棋盘怎么表示？
   
 用 Cell g_board[MAX_ROWS][MAX_COLS]，每个格子保存是否是雷、是否打开、是否标旗、周围雷数。

3. 地雷怎么放的？

 初始化时用 rand() 随机选位置，重复就重选，直到放够 g_numMines 个。 
 
4. 周围雷数怎么计算？
 
 放完雷后遍历每个格子，再检查它周围 8 个方向，把雷的数量统计到 neighborMines。

5. 点击空白格后为什么会展开一大片？

 我用递归实现了“自动展开”：如果当前格周围雷数为 0，就继续递归打开它周围的格子。

6. stb_image是什么？

 它是单文件图片加载库，方便直接读取 jpg/png，再转成 Windows 的 HBITMAP 来显示背景和图标。
 
7. 背景图是怎么显示的？
   
 先用 stbi_load 读出像素，再转成 HBITMAP，最后在 WM_PAINT 里用 StretchBlt 绘制到窗口上。

8. 胜负怎么判断？
   
 点到雷就失败；当所有非雷格都打开后就胜利。
 
9. 存档是怎么做的？
    
 直接把当前棋盘状态、行列数、雷数、游戏状态写入 save.dat，读取时再恢复。

10. 这个实现有什么不足？

 首开不安全；展开用了递归，极端情况下可能有栈风险；存档恢复时没有重新校验图片和窗口状态。 
 
11. 为什么使用stb，怎么使用stb渲染背景图片的？

 为什么用它：
简单：只要一个头文件，不用额外配置复杂库。方便：直接支持常见图片格式。
适合课程设计：能快速完成背景图、旗子、地雷图标的加载，减少自己写图片解析的工作量。
 在我的代码里，流程是：
1）stbi_load() 读取图片；
2）CreateBitmapFromPixels() 把像素转成 HBITMAP；
3）DrawImageToDC() 用 StretchBlt 画到窗口上。
 渲染流程：在 WinMain 加载图片，DrawImageToDC 用 GDI StretchBlt 绘制背景/图标，WM_PAINT/DrawBoard 在不同游戏状态下调用。

12. 如果在程序里新实现首开安全，要怎么做？

 实现首开安全最稳当的做法是“延迟放雷（deferred placement）”：不在 InitGame 放雷，而在第一次点击格子时按首点及其 8 邻域排除后再随机放雷并计算邻雷数。优点：简单、无重试、保证首点与邻居无雷；配合现有自动展开逻辑即可正常工作。

13. 使用了递归，其中的栈风险怎么处理？

 更稳妥的做法是改成队列的 BFS 或显式栈的迭代 DFS，把递归调用改为循环，这样就不会占用系统调用栈，安全性更高。
 实现思路：
1）发现 neighborMines == 0 的格子后，把它压入 queue
2）循环取出格子，展开并继续把相邻空白格入队
