# 扫雷游戏（Minesweeper）答辩问题准备——优化版
──────────────────────────────────────────
## 一、项目整体设计与架构
### Q1: 这个项目的整体架构是怎样的？
采用 WinAPI + GDI 单文件设计，核心分为四层：
层次
实现
职责
窗口框架
WinMain + WndProc 消息循环
注册窗口类、创建窗口、分发消息
游戏逻辑
Cell g_board[][] + InitGame/RevealCell/CheckWinCondition
雷区数据结构、点击展开、胜负判定
渲染层
DrawBoard/DrawImageToDC + GDI 函数
格子绘制、背景/图标贴图、文字输出
持久化
SaveGame/LoadGame + fwrite/fread
二进制存档读写

程序流程：封面 → 难度选择 → 游戏进行 → 胜负结束（可按键 R 返回难度选择），通过 GameState 枚举（STATE_COVER / STATE_DIFFICULTY / STATE_PLAYING）控制各阶段的 UI 显示与交互行为。
### Q2: 为什么使用全局变量而不是面向对象的类来组织代码？
本课程设计侧重算法实现和 WinAPI 基础，全局变量 + 函数式风格在这种单文件场景下足够清晰。Cell 结构体已经封装了格子的属性和行为，如果将整个游戏和数据拆分为 Game 类，将更具工程性——这也是我在答辩中可以提及的改进方向。
### Q3: 三个游戏状态（GameState）分别做什么？为什么这样设计？
enum GameState {
    STATE_COVER,       // 封面：显示背景图，点击任意位置进入难度选择
    STATE_DIFFICULTY,  // 难度选择：显示4个按钮，已隐藏游戏操作按钮
    STATE_PLAYING      // 游戏进行：显示棋盘 + 操作按钮（重开/存档/退出）
};
设计目的：将不同阶段的 UI 布局和交互逻辑解耦。ShowDifficultyScreen() 和 AdjustGameWindowAndButtons() 两个函数负责状态切换时的窗口/按钮显隐和尺寸自适应。WM_PAINT、WM_LBUTTONDOWN、WM_KEYDOWN 等消息处理中均根据当前状态分支执行不同逻辑。

## 二、数据结构与核心算法
### Q4: 棋盘是如何表示的？
使用二维 `Cell` 结构体数组：
struct Cell {
    bool isMine;         // 是否是地雷
    bool isRevealed;     // 是否已翻开
    bool isFlagged;      // 是否已插旗
    int  neighborMines;  // 周围 8 格的地雷数量
};

Cell g_board[MAX_ROWS][MAX_COLS];  // 固定上限 50×50，实际尺寸由 g_rows/g_cols 控制
每个格子 4 个字段恰好覆盖扫雷的全部状态需求，内存占用小（每格约 16 字节），用 bool 节省空间，int 存储 0-8 的邻雷数。
### Q5: 地雷如何随机放置？
在 InitGame() 中使用经典的 拒绝采样（rejection sampling）：
int placed = 0;
while (placed < g_numMines) {
    int r = rand() % g_rows;
    int c = rand() % g_cols;
    if (!g_board[r][c].isMine) {   // 避免重复放置
        g_board[r][c].isMine = true;
        placed++;
    }
}
用 srand(time(NULL)) 做种子。效率上，当雷密度不高时拒绝次数很少；极端情况下若雷数接近总格数的一半，碰撞会增多，但仍可控。
### Q6: 周围雷数（neighborMines）如何计算？
遍历所有非雷格子，检查其周围 8 个方向（dr=-1~1, dc=-1~1），统计邻格中 isMine == true 的数量。核心双重循环保证边界安全（nr >= 0 && nr < g_rows），这是一个典型的卷积/滑动窗口思想在离散网格上的应用。
### Q7: 点击空白格后为什么会"自动展开一大片"？展开算法是如何实现的？
使用递归深度优先搜索（DFS Flood Fill）：
static void RevealCell(int r, int c) {
    if (越界 || 已翻开 || 已插旗) return;   // 终止条件
    g_board[r][c].isRevealed = true;
    if (isMine) { g_gameOver = true; return; }
    if (neighborMines == 0) {
        // 对周围 8 个方向递归展开
        for (int dr = -1; dr <= 1; ++dr)
            for (int dc = -1; dc <= 1; ++dc)
                if (dr != 0 || dc != 0)
                    RevealCell(r + dr, c + dc);
    }
}
当点击到 neighborMines == 0 的空格时，自动递归展开所有相邻的安全空白区域，直到遇到有数字的边界格子为止。这是扫雷的核心体验逻辑。
### Q8: 胜负条件如何判定？
失败：RevealCell 翻开地雷时立即置 g_gameOver = true
胜利：CheckWinCondition() 遍历统计 isRevealed && !isMine 的格子数，若等于 总格数 - 雷数 则判定胜利
胜利判断在每次成功翻开（非雷）后触发，时间复杂度 O(R×C)，对于 50×50 的棋盘完全可接受。

## 三、图形渲染与图片加载
### Q9: 为什么使用 stb_image 库？它的优缺点是什么？
stb_image 是 Sean Barrett 开发的单头文件公共域图像库。选择理由：
零依赖：只需 #define STB_IMAGE_IMPLEMENTATION + #include "stb_image.h" 即可
格式支持广：JPG、PNG、BMP、TGA 等常见格式
适合课程项目：不需要配置 libpng/libjpeg 等复杂外部库
代价：解压大图时性能低于专用库，且所有代码集中在一个头文件中，编译稍慢。
### Q10: 从图片文件到窗口显示，完整渲染管线是怎样的？
共三步：
第一步——像素读取（LoadImageTo）：
unsigned char* data = stbi_load(path, &width, &height, &channels, 4);
// 强制要求 4 通道（RGBA），channels 参数传入 4
第二步——像素转 HBITMAP（CreateBitmapFromPixels）：
// 1. 设置 BITMAPINFOHEADER，高度取负值表示 top-down DIB
bmi.bmiHeader.biHeight = -height;   // 负值：第一行是图像顶部
// 2. CreateDIBSection 创建设备无关位图，直接获取像素缓冲区指针
HBITMAP hBitmap = CreateDIBSection(hdc, &bmi, DIB_RGB_COLORS, &pDest, NULL, 0);
// 3. 逐像素做 BGR → RGB 通道交换
pDst[i*4+0] = data[i*4+2];  // B → R
pDst[i*4+1] = data[i*4+1];  // G → G
pDst[i*4+2] = data[i*4+0];  // R → B
第三步——绘制到窗口（DrawImageToDC）：
HDC memDC = CreateCompatibleDC(hdcDest);          // 创建内存 DC
SelectObject(memDC, img->hBitmap);                 // 选入位图
StretchBlt(hdcDest, x, y, w, h, memDC, 0, 0,      // 拉伸绘制
           img->width, img->height, SRCCOPY);
DeleteDC(memDC);                                   // 释放内存 DC
为什么需要 BGR ↔ RGB 转换？ stb_image 返回标准 RGB(A) 顺序，而 Windows GDI 的 32 位位图在内存中按 BGRA 顺序存储，不转换会导致红蓝通道颠倒。
### Q11: WM_PAINT 消息中，三个游戏状态分别绘制什么内容？
状态
背景
额外内容
STATE_COVER
g_bg（start.jpg）铺满客户区
若加载失败则显示灰色背景 + 提示文字
STATE_DIFFICULTY
g_gameBg（Background.png）
标题文字"请选择游戏难度"(30号加粗字体) + 4个按钮

DrawBoard() 的绘制细节：
未翻开格：灰色实心矩形 + 黑色边框
已翻开雷格：红色背景 + 地雷图标（g_mineImg）或 * 回退
已翻开数字格：浅灰背景 + 蓝色数字（'0'+count）
插旗格：旗帜图标（g_flagImg）或红色 F 回退
游戏结束信息：胜利绿色/失败红色文字，显示在棋盘下方

## 四、交互与事件处理
### Q12: Windows 消息循环是如何工作的？
// WinMain 中
MSG msg = {};
while (GetMessage(&msg, NULL, 0, 0)) {
    TranslateMessage(&msg);   // 将按键消息转换为字符消息
    DispatchMessage(&msg);    // 分发到 WndProc 回调
}
GetMessage 是阻塞式的，无消息时线程休眠不消耗 CPU。WndProc 回调函数处理以下关键消息：
消息
处理逻辑
WM_CREATE
创建所有按钮控件（重开/存档/退出/难度选择/读档）
WM_COMMAND
响应按钮点击，分发到对应功能函数
WM_PAINT
根据 g_gameState 绘制不同界面
WM_LBUTTONDOWN
左键翻开格子 → RevealCell + CheckWinCondition
WM_RBUTTONDOWN
右键切换插旗状态
WM_KEYDOWN
封面按任意键进入难度选择；游戏结束按 R 返回
WM_DESTROY
发送 WM_QUIT 退出消息循环

### Q13: 左键单击与右键标旗的具体判断流程是怎样的？
左键翻开（WM_LBUTTONDOWN）：
1. 检查游戏状态是否为 STATE_PLAYING 且未结束
2. 从 lParam 提取鼠标坐标（LOWORD/HIWORD）
3. 通过 (x - OFFSET_X) / CELL_SIZE 反算格子的行列索引
4. 边界检查后调用 RevealCell，非雷则继续调用 CheckWinCondition
5. InvalidateRect 触发重绘
右键标旗（WM_RBUTTONDOWN）：
1. 同样反算坐标 → 边界检查
2. 仅在格子未翻开时切换 isFlagged 状态（取反）
3. 标旗的格子在 RevealCell 中被保护，不会被翻开
### Q14: 窗口尺寸如何动态适配不同难度？
AdjustGameWindowAndButtons() 函数：
按钮位置 = OFFSET_X + g_cols * CELL_SIZE + 50，始终位于棋盘右侧
窗口宽度 = max(按钮X + 150, 600)，高度 = max(棋盘高 + 150, 500)
通过 AdjustWindowRect 将客户区尺寸换算为包含标题栏/边框的窗口尺寸
SetWindowPos 调整窗口，InvalidateRect 触发重绘
### Q15: 为什么窗口创建时去掉了 WS_THICKFRAME 和 WS_MAXIMIZEBOX？
WS_OVERLAPPEDWINDOW ^ WS_THICKFRAME ^ WS_MAXIMIZEBOX
这是用位异或去掉标准重叠窗口中的可调整边框和最大化按钮。原因：棋盘尺寸由难度决定，固定大小窗口避免了用户拉伸窗口后需要处理棋盘缩放/对齐的额外复杂性。

## 五、存档与持久化
### Q16: 存档和读档是如何实现的？
采用原始二进制序列化，直接将内存数据写入文件：
// 存档——写入顺序：
fwrite(&g_rows, sizeof(g_rows), 1, fp);      // 行数
fwrite(&g_cols, sizeof(g_cols), 1, fp);      // 列数
fwrite(&g_numMines, sizeof(g_numMines), 1, fp); // 雷数
fwrite(&g_board, sizeof(g_board), 1, fp);    // 整个棋盘数组
fwrite(&g_gameOver, sizeof(g_gameOver), 1, fp); // 游戏结束标志
fwrite(&g_gameWon, sizeof(g_gameWon), 1, fp);   // 胜利标志
优点：代码极简，读写速度快（一次 fwrite 写整个数组）
局限：Cell 中的 bool 在不同编译器/平台可能有不同内存布局，跨平台兼容性差；无文件头/魔数/校验和，损坏文件难以检测
改进方向：加文件头标记（如魔数 0x4D534157）、加 CRC32 校验、使用结构化格式（JSON/二进制协议）
### Q17: 读档后如何恢复游戏状态？
LoadGame 按写入顺序 fread 恢复所有数据后，调用 AdjustGameWindowAndButtons() 重新配置窗口尺寸和按钮布局，无需重新 InitGame——因为 g_board 已经包含了完整的雷区数据。

## 六、存在的不足与改进方案
### Q18: 当前实现存在哪些不足？
不足
具体表现
改进方案
首开不安全
第一次点击可能直接踩雷
延迟放雷：首点后再随机布雷，排除首点及 8 邻域
递归栈风险

### Q19: 如果实现"首开安全"（first-click-safe），具体怎么做？
采用延迟布雷（deferred mine placement）策略：
1. InitGame 不再调用 srand，仅初始化空白棋盘
2. 首次点击时记录坐标 (firstR, firstC)
3. 随机布雷时排除 (firstR, firstC) 及其周围 8 格共最多 9 格
4. 布完雷后再计算 neighborMines
5. 正常执行 RevealCell(firstR, firstC) 的展开逻辑
这样首点及邻居绝对安全，配合已有的自动展开可保证良好体验。
### Q20: 递归展开的栈风险具体怎么解决？
改为队列 BFS（广度优先搜索）：
// 核心思路：
queue<pair<int,int>> q;
q.push({r, c});
while (!q.empty()) {
    auto [cr, cc] = q.front(); q.pop();
    if (越界 || 已翻开 || 已插旗) continue;
    g_board[cr][cc].isRevealed = true;
    if (g_board[cr][cc].neighborMines == 0) {
        for (8 个方向)
            q.push({cr+dr, cc+dc});
    }
}
也可用 std::vector 模拟显式栈做迭代 DFS，均比递归安全且无栈溢出风险。

## 七、技术细节追问
### Q21: CreateDIBSection 中 BITMAPINFOHEADER 的 biHeight 为什么设为负值？
biHeight = -height 表示自上而下（top-down）DIB——像素数据的第一行对应图像顶部。若 biHeight 为正值，则是自下而上（bottom-up）DIB，第一行对应图像底部。设为负值是为了与 stb_image 返回的自上而下像素数据直接对齐，避免额外的行翻转操作。
### Q22: 为什么 DrawBoard 中每个格子都要 CreateSolidBrush 再 DeleteObject？这不会影响性能吗？
确实会产生大量 GDI 对象创建/销毁开销。优化方案：
预创建 4 种画刷（未翻开/已翻开普通/已翻开雷/空白），在函数外复用
或使用 GetStockObject 获取系统预定义画刷
当前写法虽然性能不佳，但对于 30×16=480 个格子的规模，单次 WM_PAINT 开销仍在 1ms 量级，人眼无感知。
### Q23: 为什么 SetFocus(hwnd) 被频繁调用？
按钮点击后焦点会转移到按钮控件上，此时键盘消息将发送给按钮而非主窗口。SetFocus(hwnd) 将焦点归还给主窗口，确保 WM_KEYDOWN（如游戏结束时按 R 键重开）能正常被 WndProc 捕获。
### Q24: 程序为什么使用单文件结构？有没有考虑过分文件？
本课程设计为中等规模（约 520 行），单文件便于提交和演示。若进一步扩展，合理拆分应为：
main.cpp — WinMain 入口
game.h/cpp — 游戏核心逻辑（Cell, InitGame, RevealCell 等）
render.h/cpp — 渲染相关（DrawBoard, DrawImageToDC 等）
resource.h/cpp — 图片加载与管理

## 八、专业术语解释（Glossary）
### Windows API / WinAPI
微软 Windows 操作系统的原生编程接口，提供窗口创建、消息处理、图形绘制（GDI）等功能的 C 语言函数集。本项目中所有窗口按钮创建、事件响应、图形绘制均通过 WinAPI 实现。
### GDI (Graphics Device Interface)
Windows 的图形设备接口，负责在屏幕/打印机等输出设备上绘制图形。本项目使用 GDI 函数如 FillRect、FrameRect、StretchBlt、TextOut 等完成棋盘和文字的绘制。
### HBITMAP
GDI 中位图句柄（Handle to Bitmap），是 Windows 管理位图资源的引用标识。CreateDIBSection 和 CreateCompatibleBitmap 等函数返回 HBITMAP，用于存储和绘制图像数据。
### HDC (Handle to Device Context)
设备上下文句柄，是 GDI 绘制的核心抽象——所有绘图操作都在 DC 上进行。BeginPaint 获取窗口的绘图 DC，CreateCompatibleDC 创建内存 DC 用于离屏渲染。
### DIB (Device Independent Bitmap)
设备无关位图，用标准化的颜色格式存储像素数据，不依赖于特定显示设备。CreateDIBSection 创建的位图即为 DIB，其像素数据可被 CPU 直接读写。
### BITMAPINFOHEADER
描述 DIB 尺寸和颜色格式的结构体，包含宽度、高度、色深、压缩方式等字段，是创建 DIB 位图时必须填写的参数。
### StretchBlt
GDI 的拉伸位块传输函数，将源位图按指定尺寸缩放后绘制到目标 DC。本项目用它实现图片的缩放显示。
### WM_PAINT
Windows 消息之一，当窗口客户区需要重绘时由系统发送。应用程序在 WM_PAINT 处理函数中执行所有界面绘制逻辑。
### WM_LBUTTONDOWN / WM_RBUTTONDOWN
Windows 鼠标消息，分别在鼠标左键按下和右键按下时触发。本项目中左键用于翻开格子，右键用于标旗/取消标旗。
### 消息循环 (Message Loop)
Windows GUI 程序的核心驱动机制——GetMessage 从线程消息队列中取出消息，DispatchMessage 将其分发给 WndProc 回调处理。这个循环持续运行直到收到 WM_QUIT 退出。
### WndProc (Window Procedure)
窗口过程的回调函数，是 Windows 程序中处理所有窗口消息的集中入口。采用 switch-case 结构分发不同消息到各自的处理逻辑。
### Flood Fill（泛洪填充）
一种在离散网格上"填充连通区域"的算法。扫雷中，从点击的空格出发，递归/迭代地展开所有相邻且安全的格子，直到遇到有数字的边界。本质是图的连通分量遍历。
### DFS (Depth-First Search) / BFS (Breadth-First Search)
DFS（深度优先搜索）：沿一条路径走到尽头再回溯，本项目的递归 RevealCell 即采用此策略
BFS（广度优先搜索）：逐层扩展，用队列实现，是避免栈溢出的替代方案
### 拒绝采样 (Rejection Sampling)
一种随机采样方法：在候选空间中随机取点，若不满足条件则拒绝并重试，直到获得足够样本。本项目布雷即采用此策略——随机选格子，若已有雷则重选。
### stb_image
Sean Barrett 开发的单头文件 C 图像加载库（公共域），支持 PNG、JPG、BMP 等多种格式。使用时只需 #define STB_IMAGE_IMPLEMENTATION 后 #include "stb_image.h"，无需链接外部库。API 核心函数：
stbi_load(path, &w, &h, &channels, desired_channels) — 加载图像，返回像素数组
stbi_image_free(data) — 释放像素内存
### 二进制序列化 (Binary Serialization)
将程序中的数据结构直接以内存中的二进制形式写入文件，读回时按同样顺序还原。优点是速度快、代码简单；缺点是缺乏可移植性和可读性。
### 延迟布雷 (Deferred Mine Placement)
一种扫雷实现技术：不在初始化时布雷，而在玩家首次点击后再随机放置地雷，并排除首点及其邻域，从而保证首点绝对安全。
### CreateDIBSection
GDI 函数，创建 DIB 位图并同时返回像素数据的直接内存指针（ppvBits），允许程序直接读写位图像素，非常适合与 stb_image 这类像素级库对接。
### AdjustWindowRect
WinAPI 函数，根据指定的客户区（client area）矩形尺寸，反算包含标题栏、边框在内的窗口（window）总尺寸。创建/调整窗口时使用，确保客户区刚好容纳所需内容。
### InvalidateRect
通知 Windows 某窗口的指定矩形区域"已失效"需要重绘。系统在下一次消息循环空闲时发送 WM_PAINT 消息。本项目中每次操作后调用它触发界面刷新。
### 魔数 (Magic Number)
文件格式中用于标识文件类型的固定字节序列（如 PNG 文件头 \x89PNG）。在存档文件中加入魔数可以快速识别是否为有效存档，防止读取错误文件。
